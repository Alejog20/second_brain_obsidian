## Folder structure
Folders are whatever topic areas this vault's owner actually uses - the pipeline discovers
them from what's really there (`agents_md_builder.py`'s taxonomy scan), it doesn't enforce a
fixed numbering scheme. The generated section below always lists exactly what that scan
found for this vault; expect something in this shape:
- `<Topic>/` — one folder per subject area (e.g. `AI/`, `Security +/`, `Linux/`), freeform,
  named by whoever writes the vault, not the pipeline
- Daily notes — live at the vault root by default (`vault.daily_notes_folder: ""` in
  `config.yaml`), or in a dated subfolder if that's set to one; named per
  `vault.daily_note_date_format` (e.g. `2026-07-20.md`)
- `_reports/` — nightly output: `Review-{date}.md` (the audit log) and `Recap-{date}.md`
  (a short reinforcement summary of what got digested that day)
- `_staging/` — proposed edits awaiting approval (dry-run mode only)

## Naming and linking conventions
- Daily notes: named per `vault.daily_note_date_format` in `config.yaml` - don't assume a
  specific format (`MM-DD-YYYY` and `YYYY-MM-DD` are both in real use across vaults this
  pipeline runs against); check `config.yaml` rather than guessing
- Atomic/topic notes: descriptive title case, no dates in the filename
- Every atomic note the pipeline creates includes a `source` field in frontmatter
  (e.g. `source: "[[2026-07-20]]"`), linking back to the daily note it was distilled from
- Prefer linking over duplicating: before creating a new note, check for
  an existing note on the same concept and link to it instead of forking it

## Frontmatter schema
```yaml
title:
created: YYYY-MM-DD
tags: []
source: "[[<daily note date>]]"   # present only on notes distilled from a daily note
status: draft | reviewed
```

## AI-added content
Some notes carry an `> [!ai-fact-check]` callout near the end - the pipeline appends these
automatically after fact-checking that day's new notes against live web search, and after
adding a simple example where a concept was stated without one. Callouts are always appended,
never merged into the user's own prose, so they stay visually and structurally separate from
what the user actually wrote:
- Treat a callout's content as a note, not a correction already applied above it - the user
  hasn't necessarily reviewed or acted on it yet.
- Never delete or edit an `> [!ai-fact-check]` callout; only the nightly pipeline writes these.
- If a callout and the note body disagree, say so plainly rather than silently picking one.

## Token-saving tips for AI tools
- This file is the map - read it before opening notes, not instead of grep/search on demand.
- Prefer a targeted read of the specific note(s) a question is about over scanning a whole
  folder; the folder/tag vocabulary below is usually enough to know where to look first.
- Don't re-summarize a note's full content back to the user if a short answer plus a
  `[[link]]` to the source note will do.

## How to answer questions
- Answer only from what's in this vault. If it isn't covered, say so
  directly — don't fill the gap with general knowledge presented as if
  it came from the notes.
- Cite the specific note(s) as `[[wikilinks]]`, not "according to my search."
- No filler: skip "great question," "I'd be happy to," disclaimers, and
  hedging. State the answer, then the source.
- If two notes disagree, say that plainly and name both — don't silently
  prefer the newer one.
- Register: [fill in — e.g. "direct, technical, no marketing language,
  short sentences"]

## What not to do
- Don't invent frontmatter fields, tags, or folders outside this schema.
- Don't rewrite a note's substance to "improve" it — flag unclear notes
  in the nightly report instead of silently rewording half-formed ideas.
- Don't merge two notes without listing it as a significant change in
  the nightly report.
